# Database

## Migrations

Never modify an existing migration that has been applied to any environment. Always create a new migration for schema changes. Treat applied migrations as immutable history.

```bash
# Generate from model changes — always review the output
alembic revision --autogenerate -m "add_email_index_to_users"

# Apply
alembic upgrade head

# Rollback one step
alembic downgrade -1
```

Review autogenerated migrations before committing. Alembic cannot detect renamed columns (it sees drop + add), index changes on expressions, or enum value modifications. Write those by hand.

## Query Patterns

Use the 2.0-style `select()` API, not the legacy `session.query()`:

```python
# Good — 2.0 style
stmt = select(User).where(User.is_active == True).order_by(User.created_at.desc())
result = await session.execute(stmt)
users = result.scalars().all()

# Bad — legacy query API
users = session.query(User).filter(User.is_active == True).all()
```

Avoid N+1 queries. Use `joinedload` for single related objects, `selectinload` for collections:

```python
stmt = (
    select(Order)
    .options(joinedload(Order.user), selectinload(Order.items))
    .where(Order.id == order_id)
)
```

Never call `session.execute` inside a loop. Batch with `where(Model.id.in_(ids))` instead.

## Transactions

Use async sessions as context managers. Keep transactions short — gather data outside, wrap only writes.

```python
async with async_session() as session:
    async with session.begin():
        session.add(user)
        session.add(audit_log)
    # commit happens automatically when the begin() block exits
```

Never do HTTP calls, file I/O, or slow external work inside an open transaction. If you need to call an external service and then write, do the call first, then open a transaction for the write.

## Connection Pooling

Configure pool size and overflow in `create_async_engine`:

```python
engine = create_async_engine(
    settings.database_url,
    pool_size=10,
    max_overflow=20,
    pool_pre_ping=True,
    pool_recycle=3600,
)
```

`pool_size` is the number of persistent connections. `max_overflow` allows bursts beyond that. `pool_pre_ping` detects stale connections before use. `pool_recycle` prevents connections from living too long (important behind load balancers that drop idle connections). Match `pool_size` to your expected concurrent request count, not to the database's `max_connections`.
