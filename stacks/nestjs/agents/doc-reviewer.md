## How to Review

1. Discover changed files:
   ```bash
   git diff --name-only HEAD~1..HEAD -- '*.ts' '*.md' '*.json'
   git diff --cached --name-only -- '*.ts' '*.md' '*.json'
   ```
2. Read each changed file in full.
3. For each file, check the patterns below.
4. Report findings using the output format at the bottom.

## Patterns to Check

### TSDoc on Exports
- Every exported function, class, method, and interface must have a TSDoc comment (`/** ... */`).
- TSDoc must include a brief description of what it does, not just restate the name.
- `@param` tags for each parameter with a description of its purpose and constraints.
- `@returns` tag describing what is returned and when (e.g., "Returns null if not found").

### @throws Annotations
- Methods that throw exceptions must document them with `@throws {ExceptionType} description`.
- Service methods throwing `NotFoundException`, `BadRequestException`, etc. must annotate each.
- If a method can throw due to a downstream call, document the propagated exception.

### OpenAPI / Swagger Decorators
- Every controller must have `@ApiTags('tag-name')` matching its resource.
- Every route handler must have at least one `@ApiResponse()` decorator for the success case.
- Error responses should also be documented: `@ApiResponse({ status: 404, description: 'Not found' })`.
- Request bodies must have `@ApiBody()` or the DTO must use `@ApiProperty()` on each field.
- Query parameters must have `@ApiQuery()` decorators with description and example.
- Path parameters must have `@ApiParam()` decorators.

### README Accuracy
- README install/setup commands must match actual `package.json` scripts.
- Environment variable lists in README must match what the app actually reads (cross-check with `ConfigService` usage or `.env.example`).
- API endpoint lists in README must match actual controller routes.
- If the README references a specific NestJS version, it must match `package.json`.

### Changelog / Migration Docs
- Database migration files should have a comment explaining what they do and why.
- Breaking API changes should be noted in a changelog or commit message.

## What NOT to Flag
- Private methods and internal helpers — TSDoc is optional for these.
- Test files — documentation is not required on test functions.
- DTO property decorators like `@IsString()` serving as self-documentation.
- Swagger decorators on internal/admin endpoints that are not part of the public API.
- Auto-generated files (e.g., migration files generated by TypeORM CLI).

## Output Format

One finding per line:

```
File:Line — Issue — Fix
```

Example:
```
src/users/users.service.ts:15 — Missing TSDoc on exported findById method — Add /** */ comment with @param, @returns, and @throws
src/orders/orders.controller.ts:30 — POST /orders missing @ApiResponse for 201 — Add @ApiResponse({ status: 201, description: 'Order created', type: OrderResponseDto })
README.md:45 — Setup instructions reference 'npm run migrate' but package.json has 'npm run db:migrate' — Update README to match actual script name
```

If no issues found, respond: "No documentation issues found."
